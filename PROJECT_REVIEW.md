Дата ревью: 2026-06-05

> Цель документа — не похвалить проект, а максимально честно показать слабые места, риски, недоказанные гипотезы и точки кратного роста.

---

## 0. Executive summary

Проект сейчас выглядит как *персональный cockpit для распределения будущей зарплаты*, сделанный под очень конкретный профиль: молодой пользователь/студент с доходом около 25 000 грн, низкими обязательными расходами, большим количеством желаний и желанием заранее расставлять приоритеты.

Главная сила идеи — не в очередном трекере расходов, а в сдвиге фокуса: *не “куда ушли деньги”, а “как заранее принять решение, куда они пойдут”*. Это хороший угол. Но в текущем виде продукт находится в опасной зоне: он уже сложнее заметок/таблицы, но ещё недостаточно автоматизирован, социально понятен и доверителен, чтобы заменить полноценные финтех-приложения или банковские сценарии.

Ключевой риск: продукт может оказаться слишком персональным инструментом автора, а не масштабируемым продуктом. Большая часть текущих фичей полезна при высокой мотивации пользователя вручную вести данные. Для массового пользователя это слабое место: если данные надо много вводить руками, retention почти всегда падает.

## 1. Ценность продукта

### 1.1. Какую проблему решает проект

Проект решает проблему *предварительного распределения ограниченного дохода* между:

- обязательными расходами;
- резервом/страховкой;
- инвестициями;
- желаниями;
- долгосрочными целями;
- эмоциональными/импульсивными покупками;
- “капитальными” покупками, которые могут улучшить будущую траекторию.

Это не классический expense tracker. Продукт ближе к:

- бюджетированию по методу envelope budgeting;
- wish queue / purchase prioritization;
- decision journal для финансовых решений;
- lightweight financial planning для людей без сложного капитала.

### 1.2. Насколько проблема реально важна

Проблема реальна, но важность зависит от сегмента.

*Для кого боль сильная:*

1. Люди с фиксированной зарплатой и частыми “хочу купить”.
2. Студенты/молодые специалисты, у которых доход ещё небольшой, а желаний много.
3. Пользователи, которые пытаются не жить “от импульса до импульса”.
4. Люди без привычки инвестировать/копить, которым нужен простой ритуал перед зарплатой.
5. Пользователи с тревогой вокруг денег: им важна ясность до того, как деньги пришли.

*Для кого боль слабая:*

1. Пользователи с уже настроенными банковскими категориями и автоплатежами.
2. Люди с высоким доходом, где мелкие желания не требуют очереди.
3. Семейные бюджеты: текущая модель слишком индивидуальная.
4. Люди, которым нужно строгое бухгалтерское отслеживание фактических расходов.
5. Пользователи, которые не готовы вводить желания, суммы, приоритеты и дедлайны вручную.

### 1.3. Главный value proposition сейчас

Текущий value proposition можно сформулировать так:

> “Capital Queue помогает до зарплаты решить, что купить, что отложить и сколько оставить на стабильность/инвестиции, чтобы не тратить деньги импульсивно”.

Это понятнее, чем “personal finance cockpit”. Термин cockpit звучит красиво, но для массового пользователя может быть слишком абстрактным.

### 1.4. Есть ли более простые способы решить проблему

Да. И это самый опасный конкурентный фактор.

Пользователь может заменить продукт:

1. Таблицей Google Sheets / Notion.
2. Заметкой “что купить после зарплаты”.
3. Банковскими “банками”, “конвертами”, “копилками”.
4. Простым правилом: 50/30/20.
5. Todo-list с ценами.
6. Telegram-ботом с кнопками “купить/отложить”.
7. Функцией wish list в маркетплейсе.
8. Автопереводом на отдельные счета после зарплаты.

Чтобы выигрывать, продукт должен давать не просто форму для списка желаний, а *решение, которого сложно добиться в таблице*: умная приоритизация, психологическая защита от импульса, прогноз regret, сценарии, автоматические правила, интеграции, история решений и обучение пользователя.

### 1.5. Слабые места ценности

1. *Слишком много ручного ввода.* Желания, суммы, приоритеты, оценки, накопления, портфель, кошельки — всё требует дисциплины.
2. *Непонятно, почему пользователь должен возвращаться каждую неделю.* Сейчас главный ритуал — зарплата/месяц. Это может быть слишком редкая частота для retention.
3. *Нет доказанного “aha moment”.* Что пользователь должен почувствовать за первые 3 минуты? “Я понял, что могу купить X, а Y лучше отложить”? Это надо усилить.
4. *Нет сильной автоматизации фактических расходов.* Продукт планирует, но не проверяет, выполнил ли пользователь план.
5. *Сегмент слишком узко описан в README.* “Студент, 25 000 грн/мес, живёт с родителями” — хорошо для MVP, но плохо для внешнего позиционирования.
6. *Риск морализаторства.* Категории вроде `dopamine`, `waste`, “утечки” могут помогать дисциплине, но могут вызывать сопротивление: пользователь не хочет, чтобы приложение осуждало его желания.

## 2. Конкурентный анализ

### 2.1. Прямые и косвенные аналоги

#### Персональное бюджетирование

- YNAB — zero-based budgeting, сильная методология, высокая retention за счёт привычки.
- EveryDollar — простое бюджетирование по категориям.
- Monarch Money — агрегатор финансов, семейный бюджет, цели.
- Copilot Money — автоматическая категоризация, красивый UX.
- PocketGuard — “сколько можно потратить”.
- Goodbudget — envelope budgeting.
- Spendee — бюджеты, кошельки, shared finance.

#### Банковские продукты

- Monobank jars / банки.
- Revolut pockets/vaults.
- Wise jars.
- Приват/другие банковские копилки.
- Автопереводы на накопительные счета.

#### Таблицы и no-code

- Google Sheets budget templates.
- Notion finance templates.
- Airtable/Сoda personal finance dashboards.

#### Wish list / goal tracking

- Wishlists в маркетплейсах.
- Todoist/Notion lists с ценами.
- Goal trackers / savings apps.

#### AI-finance ассистенты

- Cleo.
- ChatGPT custom GPT для бюджета.
- Banking AI assistants.
- Финансовые коучи/боты.

### 2.2. В чем проект проигрывает конкурентам

1. *Нет банковских интеграций.* Конкуренты автоматически видят транзакции, доходы, категории, регулярные платежи.
2. *Нет мобильного native UX.* PWA — неплохо, но финансовые привычки часто живут в телефоне, уведомлениях и виджетах.
3. *Нет семейных/совместных сценариев.* Многие платные пользователи в personal finance — пары и семьи.
4. *Нет trust layer.* Финансовые данные чувствительны. Нужны privacy, encryption, backup, audit, понятные гарантии.
5. *Нет методологии уровня YNAB.* Есть алгоритм приоритизации, но нет обучающей системы привычек.
6. *Нет доказанной автоматической категоризации.* Ввод и классификация вручную.
7. *Нет market-ready onboarding.* README объясняет технически, но не продаёт “первую победу” пользователя.
8. *Нет понятной монетизации.* AI опционален, но не показано, за что платить.
9. *Нет многопользовательской модели.* Текущая auth-модель — один пользователь + PIN.
10. *Слабый moat.* Vanilla app + SQLite + очередь желаний легко копируется.

### 2.3. В чем проект может выиграть

1. *Фокус на будущей зарплате, а не прошлых расходах.* Это дифференциация.
2. *Психология покупки.* Trade-off, regret, долгосрочная ценность, эмоциональный импульс — сильный угол.
3. *Очень простой self-hosted режим.* Для privacy-conscious пользователей это плюс.
4. *Локальный рынок/язык.* Русскоязычный/украинский контекст, гривны, зарплатный цикл.
5. *Скорость продукта.* Маленький стек позволяет быстро добавлять фичи.
6. *AI-коуч поверх личного плана.* Если сделать хорошо, может быть сильнее таблицы.
7. *Decision journal.* История “что я хотел, что купил, о чём пожалел” — редкая и потенциально ценная фича.

### 2.4. Потенциальная ниша

Самая перспективная ниша — не “финансовый трекер для всех”, а:

> “Антиимпульсивный планировщик зарплаты для молодых специалистов: помогает до зарплаты выбрать покупки, которые реально двигают жизнь вперёд”.

Это уже звучит конкретнее и отличает продукт от обычного budget tracker.

## 3. Пользовательский опыт (UX)

### 3.1. Что может быть непонятно пользователю

1. *Разница между слоями, категориями, типами, band и score.* Слишком много таксономии. Новый пользователь видит: слой, категория, тип must/should/nice, priority, trajectory, emotional, band, score type. Это когнитивно тяжело.
2. *Что такое “Capital Queue”.* Название красивое, но не мгновенно объясняет пользу.
3. *Что означает “траектория”.* В интерфейсе это может быть неочевидно: траектория жизни? Карьеры? Финансов?
4. *Почему “эмоциональное желание” влияет слабо.* Пользователь не понимает алгоритм без объяснения.
5. *Чем кошельки отличаются от ручного плана.* Оба распределяют деньги на месяц, но ментальная модель пересекается.
6. *Чем инвестиции как фиксированный пункт отличаются от вкладки “Инвестиции”.* В плане есть `investmentFixed`, отдельно есть портфель. Для новичка это может выглядеть дублированием.
7. *Что значит “закрыть месяц”.* Это бухгалтерский термин, но пользователю может быть непонятно, когда именно нажимать.
8. *Что делать в первый день использования.* Нет явного guided flow: “1) укажи зарплату; 2) добавь 5 желаний; 3) получи план”.
9. *Почему приложение спрашивает PIN, а не аккаунт.* Для self-hosted норм, для массового продукта странно.
10. *AI выключен по умолчанию.* Пользователь видит вкладку, но она не работает без env vars. Для продакта это разочарование.

### 3.2. Где возникают лишние действия

1. Пользователь должен вручную создать PIN.
2. Потом вручную настроить зарплату.
3. Потом вручную добавить каждое желание.
4. Потом вручную указать стоимость, тип, приоритет, категорию.
5. Потом вручную вести накопления.
6. Потом вручную вести кошельки.
7. Потом вручную вести инвестиции/оценки активов.
8. Потом вручную закрывать месяц.
9. Потом вручную импортировать/экспортировать backup.

Это много. Для MVP допустимо, для retention — риск.

### 3.3. Какие функции стоит убрать или упростить

#### Упростить в onboarding

1. Скрыть `band` от пользователя: это техническая/автоматическая классификация.
2. Скрыть `scoreType` до момента крупной покупки.
3. Не спрашивать одновременно priority, emotional, trajectory у каждой покупки. Для quick-add достаточно: название, сумма, “насколько важно”.
4. Свести категории в 4–5 понятных групп на первом экране.
5. Кошельки и ручной план объединить или чётко развести: “кошельки = категории трат”, “план = покупки”.

#### Возможно убрать из MVP

1. Ручной учёт инвестиционного портфеля с транзакциями и valuations — это отдельный продукт. Он увеличивает сложность и отвлекает от главной идеи.
2. Полная оценка покупки по 13 критериям — мощно, но тяжело. Лучше показывать только для крупных покупок.
3. История закрытых месяцев полезна, но если нет follow-up анализа, она превращается в архив.
4. Импорт/restore full backup в UI — важно технически, но в UX лучше спрятать в settings/advanced.

### 3.4. Главные UX-рекомендации

1. *Сделать первый сценарий за 2 минуты.* Зарплата → обязательные → 5 желаний → результат.
2. *Добавить “режим новичка”.* Только базовые поля. Advanced поля раскрываются позже.
3. *Ввести понятный главный экран:* “После зарплаты ты можешь безопасно купить: A, B. Лучше отложить: C. Останется: X”.
4. *Добавить объяснение алгоритма простым языком.* Не формулами, а “почему это выше в очереди”.
5. *Добавить эмоциональную обратную связь:* “Ты сохранил 3 200 грн от импульсивных покупок”.
6. *Добавить шаблоны.* “Студент”, “первый доход”, “коплю на ноутбук”, “хочу меньше импульса”.
7. *Сделать mobile-first.* Финансовые решения часто принимаются с телефона перед покупкой.
8. *Переименовать сложные сущности.* Например, `trajectory` → “двигает меня вперёд”, `leakage` → “сомнительная покупка”.

## 4. Техническая часть

### 4.1. Архитектура

Текущая архитектура:

- Backend: Express monolith в одном большом `server.js` примерно на 2000 строк.
- DB: SQLite через `better-sqlite3`.
- Frontend: один большой vanilla JS файл `public/app.js` примерно на 2000 строк.
- Domain logic частично вынесена в `src/allocation.js`, `src/categories.js`, `src/db.js`, `src/auth.js`, `src/ai.js`.
- PWA: manifest + service worker.
- CI: GitHub Actions с tests/audit/e2e/deploy to Fly.

Для персонального/self-hosted приложения это нормально. Для масштабируемого SaaS — архитектура пока не готова.

### 4.2. Архитектурные недостатки

1. *Слишком большой `server.js`.* API, backup, import/export, portfolio, auth, rate limit, AI, CSV — всё в одном файле. Это ухудшает поддержку и повышает риск регрессий.
2. *Слишком большой `public/app.js`.* Рендеринг, state management, charts, forms, assistant, sync, модалки — всё в одном файле. Сложно тестировать и развивать.
3. *Нет явного слоя сервисов.* Бизнес-операции смешаны с Express handlers.
4. *Нет схемы валидации через библиотеку.* Валидация вручную разбросана по коду. Это лучше, чем ничего, но сложно гарантировать полноту.
5. *Миграции встроены в `src/db.js`.* Для раннего этапа нормально, но при росте схемы нужен нормальный migration system.
6. *Нет typed contracts между frontend/backend.* Можно легко сломать поле `investmentFixed`/`investment_fixed`, `layer`/`bucket` и подобные совместимости.
7. *Нет разделения личного режима и SaaS-режима.* Auth, DB, backup и tenancy построены вокруг одного пользователя.
8. *Нет feature flags.* AI, investments, advanced scoring, import/export лучше включать/выключать контролируемо.

### 4.3. Потенциальные проблемы масштабирования

1. *SQLite + один пользователь.* Для self-hosting хорошо, для multi-user SaaS нужна multi-tenant модель, Postgres и нормальные миграции.
2. *Синхронный `better-sqlite3`.* В одном процессе ок, но при росте нагрузки блокирующие операции могут стать проблемой.
3. *In-memory rate limit.* При нескольких инстансах лимиты не синхронизируются. После рестарта сбрасываются.
4. *Backup файловой системы.* Для SaaS нужны object storage, encryption, retention policies, user-level export/delete.
5. *AI context строится из всех данных.* При росте объёма желаний/истории контекст будет тяжелеть и дорожать.
6. *Нет background jobs.* Обновление цен, backup, AI-операции, напоминания лучше выносить в job queue.
7. *Service worker может кэшировать устаревшие assets.* Нужно аккуратно управлять версиями кэша при деплоях.
8. *Нет observability.* Есть structured logs, но нет метрик, tracing, error reporting, product analytics.

### 4.4. Безопасность

Что уже хорошо:

- `SESSION_SECRET` обязателен в production.
- Cookie `httpOnly`, `sameSite=lax`, `secure` в production.
- Есть CSP, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`.
- PIN хешируется bcrypt.
- Есть rate limit на login, AI, import, backup.
- Express `x-powered-by` отключен.
- JSON body limit 1mb.
- `SETUP_TOKEN` поддерживается для первого PIN.

Основные риски:

1. *Auth-модель слабая для SaaS.* Один PIN без email, MFA, recovery, device/session management, password reset.
2. *PIN может быть коротким.* Минимум 4 символа — это мало для финансового приложения. Rate limit помогает, но лучше требовать 6+ цифр или passphrase.
3. *Нет CSRF-защиты.* `sameSite=lax` снижает риск, но для state-changing POST лучше добавить CSRF token, особенно если будет расширение auth.
4. *Нет encryption at rest.* SQLite и backup JSON содержат финансовые данные в открытом виде.
5. *Full export/backup отдаёт чувствительные данные.* Нужны предупреждения, возможно encrypted export.
6. *AI отправляет финансовый контекст внешнему провайдеру.* Есть предупреждение в UI, но нет granular consent, redaction, policy, audit.
7. *Логи могут содержать IP и path.* Это нормально, но нужна privacy policy, retention, scrub sensitive query params.
8. *Refresh prices вызывает внешние API по тикерам.* Нужны timeout, SSRF-анализ, allowlist формата ticker. Сейчас тикер нормализуется, но внешний fetch всё равно требует аккуратности.
9. *Import full backup потенциально опасен логически.* Валидация есть, но импорт полностью заменяет данные. Нужны preview/diff/undo.
10. *Нет session revocation.* JWT живёт 30 дней; если cookie украдена, отозвать сессию невозможно без смены секрета.

### 4.5. Производительность

1. Для одного пользователя производительность, скорее всего, достаточная.
2. Frontend генерирует HTML строками и перерисовывает view целиком. На сотнях желаний нормально, на тысячах начнёт тормозить.
3. Portfolio calculations фильтруют транзакции/оценки по каждому активу в памяти. При больших данных лучше агрегировать SQL-запросами.
4. Canvas-чарты lightweight — хорошо.
5. Нет bundle step — быстрый деплой, но нет tree-shaking/minification.
6. Service worker кэширует базовые assets, но нет продвинутой стратегии invalidation.
7. AI streaming сделан полезно, но нет cancellation/abort UX.

## 5. Монетизация

### 5.1. Насколько жизнеспособна текущая модель

В репозитории явной монетизации нет. Если продукт останется self-hosted personal tool — монетизация почти нулевая. Если делать SaaS, надо изменить архитектуру и позиционирование.

Платёжеспособность сегмента “студент с 25 000 грн/мес” ограничена. Он может любить продукт, но платить много не будет. Поэтому монетизация должна либо:

1. быть очень дешёвой и массовой;
2. продавать premium AI/automation;
3. идти в более платежеспособные сегменты: молодые специалисты, пары, фрилансеры, люди с доходом 50–150k грн/мес;
4. иметь B2B2C/education/coaching angle.

### 5.2. Возможные модели

#### Freemium

- Free: 1 зарплатный план, до 20 желаний, базовая очередь.
- Plus: unlimited wishes/history, сценарии, reminders, export/import, encrypted backups.
- AI Pro: AI-коуч, regret prediction, purchase review, weekly insights.

#### Subscription

- $2–5/мес для локального рынка.
- $8–12/мес для international version.
- Family plan: $8–15/мес.

#### One-time purchase

- Lifetime self-hosted license.
- “Personal finance cockpit template” для разработчиков/privacy users.

#### Marketplace / affiliate

Осторожно: финансовые affiliate могут конфликтовать с доверием. Но можно:

- рекомендовать savings accounts;
- брокеров;
- образовательные курсы;
- cashback cards;
- insurance products.

Только при прозрачности и без превращения продукта в рекламную машину.

#### Coaching / courses

- “30-дневный челлендж антиимпульсивных покупок”.
- Персональные финансовые разборы.
- Templates для разных жизненных сценариев.

### 5.3. Какие способы монетизации отсутствуют

1. Premium AI.
2. Paid encrypted cloud sync.
3. Mobile app subscription.
4. Family/shared budget plan.
5. Coaching upsell.
6. Community challenges.
7. Bank/broker integrations as paid add-on.
8. Advanced analytics/reporting.
9. Behavioral insights: regret tracking, impulse score.
10. White-label для финансовых консультантов.

### 5.4. Как увеличить LTV и удержание

1. Еженедельный ritual: “проверь очередь перед выходными”.
2. Push/Telegram reminders перед зарплатой.
3. “Impulse cooldown”: крупная покупка попадает в 48h waitlist.
4. “Regret review”: через 30 дней после покупки спросить, стоило ли оно того.
5. Сравнение месяцев: “в этом месяце ты направил на карьерный капитал +18%”.
6. Streak: “3 месяца без перерасхода”.
7. Цели с прогрессом и визуальным payoff.
8. AI monthly review.
9. Автоматический импорт банковских/CSV транзакций.
10. Shared accountability: друг/партнёр видит только выбранные цели.

## 9. Взгляд инвестора

### 9.1. Инвестировал бы я сейчас?

Сейчас — нет, если речь о венчурной инвестиции. Как indie/SaaS bootstrap-проект — да, можно продолжать. Как venture-scale startup — пока недостаточно доказательств.

### 9.2. Основные причины отказа

1. *Не доказан рынок.* Не видно traction, retention, платящих пользователей.
2. *Слишком узкий исходный ICP.* Студент с 25k грн — неочевидно платёжеспособный сегмент.
3. *Нет defensibility.* Идею можно скопировать; нужен data moat, behavioral model, distribution, community или bank integrations.
4. *Высокий manual input burden.* Это убивает retention у personal finance apps.
5. *Нет монетизации.* Не показано, за что пользователь платит.
6. *Нет multi-user архитектуры.* Текущий код ближе к personal tool/self-hosted app.
7. *Сильные конкуренты.* Банки и YNAB-like продукты имеют больше данных и доверия.
8. *Непонятный growth loop.* Пока нет встроенной вирусности/реферальности.
9. *Слабая trust story.* Финансовые данные требуют серьёзного privacy/security позиционирования.
10. *Фича-scope расползается.* Очередь желаний + кошельки + инвестиции + AI + backup + portfolio может потерять фокус.

### 9.3. Что изменить, чтобы проект стал инвестиционно привлекательнее

1. Доказать retention: хотя бы 100–500 активных пользователей, которые возвращаются 2–3 месяца.
2. Сформулировать узкий, но платёжеспособный ICP: например, “young professionals with first disposable income”.
3. Снизить manual input через AI/Telegram/bank import.
4. Найти сильный hook: “AI anti-impulse purchase coach”.
5. Показать willingness to pay: первые платящие подписчики.
6. Добавить уникальные behavioral insights: regret tracking, impulse fingerprint.
7. Создать growth loop: share cards, challenges, templates.
8. Построить trust story: encrypted backups, privacy-first, self-hosted/cloud choice.
9. Упростить продукт до одного core workflow: plan salary → decide purchases → review regret.
10. Показать roadmap к масштабируемой архитектуре.

## 11. Дополнительные технические замечания по коду

### 11.1. Backend

- `server.js` стоит разделить первым: это главный источник будущего технического долга.
- Rate limit через `Map` подходит только для single-instance.
- JWT без server-side session store не позволяет отзывать отдельные сессии.
- `PIN >= 4` лучше повысить до 6+ или сделать passphrase mode.
- Import full backup должен иметь preview/diff/undo.
- AI calls нуждаются в timeout/abort/retry policy.
- Нужно добавить более строгий ticker validation для price refresh.
- Backup JSON стоит шифровать хотя бы опционально.
- Нужна отдельная security policy в README.

### 11.2. Frontend

- HTML string rendering требует постоянной дисциплины с `escapeHtml`. Сейчас многие пользовательские поля экранируются, но архитектурно риск остаётся.
- Нужно вынести views из `app.js`.
- Нужно добавить accessibility checks: labels, keyboard navigation, focus states, modal trapping.
- Нужно улучшить empty states и onboarding.
- Нужно сделать mobile-first ревизию форм: на телефоне ручной ввод большого количества полей будет болезненным.

### 11.3. Data model

- `bucket` используется как слой/legacy, `layer` как новая модель. Нужна окончательная миграция терминов.
- `settings` как JSON store удобен, но при росте усложнит запросы и миграции.
- Смешение goals/item saved amounts может привести к рассинхронизации.
- Investment model уже довольно сложная для продукта, где core value — purchase prioritization.

## 12. Самый жёсткий вывод

Проект не должен пытаться стать “ещё одним финансовым приложением”. В этой категории слишком много сильных игроков, а ручной ввод убивает привычку.

Самый перспективный путь — стать *поведенческим антиимпульсивным слоем перед покупкой*: быстро добавить желание, увидеть trade-off, переждать cooldown, получить AI/алгоритмическое объяснение, а потом через месяц проверить, было ли решение хорошим.

Если продукт сфокусируется на этом, у него появится отличимость. Если продолжит расширяться во все стороны — бюджет, кошельки, инвестиции, портфель, AI, backup, сценарии — есть риск стать сложной персональной панелью, которую любит только автор.
