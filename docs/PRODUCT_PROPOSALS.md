# Продуктовые предложения по аудиту (2026-06-12)

Эти пункты аудита — не баги, а продуктовые/архитектурные решения. Чинить их «коммитом
между делом» вредно: каждый требует решения владельца и отдельной итерации.
Здесь — конкретный план по каждому, чтобы решение можно было принять быстро.

---

## [1.1] Пять продуктов сразу → иерархия «ядро / сателлиты»

**Предложение.** Ядро: план + очередь + закрытие месяца — всегда в навигации.
Сателлиты — выключаемые модули с фича-флагами в настройках (паттерн уже есть: AI):

| Модуль | Флаг (settings) | По умолчанию |
|---|---|---|
| Инвестиции | `module_investments` | вкл |
| Кошельки | `module_wallets` | вкл |
| AI-ассистент | `AI_PROVIDER` (есть) | выкл |
| Прайс-трекер по ссылкам | `module_pricecheck` | вкл |
| Monobank-сводка | `MONOBANK_TOKEN` (есть) | выкл |

Реализация: ключи в `settings`, секция «Модули» на экране настроек (уже создан),
`metaPayload()` отдаёт флаги, клиент скрывает navigation-пункты и плитки «Ещё».
Объём: ~1 день. Риск низкий. Главный выигрыш — онбординг и фокус.

## [1.2] Три механизма распределения → единый «бюджет излишка»

Видимое следствие уже закрыто (одна формула `remainingSurplus()` на Кабинете,
Плане и Кошельках). Полное слияние — отдельная модель данных:

**Предложение.** Одна сущность `allocation_entry { source: manual|wallet|goal, amount, target }`
вместо трёх таблиц-механизмов. Миграция: ручной план и кошельки переливаются в неё 1:1,
взносы в цели регистрируются как записи с `source=goal`. Один прогресс-бар
«Распределено X из Y», одна валидация перерасхода.
Объём: 2–3 дня + миграция данных. Делать только после решения по 1.1
(если кошельки станут выключаемым модулем, слияние упрощается).

## [1.3] Нет якоря «факта» → поле «факт обязательных расходов» при закрытии месяца

**Предложение (минимальная версия, 0.5 дня).**
1. В `POST /api/plan/close` — необязательное поле `actualSurvivalCost`.
2. В снимок месяца пишется дельта `actualSurvivalCost − survivalCost`.
3. История показывает «план 25 000 / факт 27 400 (+9,6%)»; три месяца подряд
   дельта >10% → инсайт «обязательные расходы занижены, обновите план».
4. Если настроен Monobank — сводка трат за месяц подставляется в поле автоматически.

Расширенная версия — импорт CSV-выписки с маппингом в категории (см. раздел 18) —
отдельная итерация после проверки, что минимальную версию вообще заполняют.

## [1.5] Single-user по дизайну → зафиксировать позиционирование, мультиюзер по триггеру

**Предложение.** Сейчас: честно позиционировать как single-user self-hosted
(как Actual Budget) — строка в README + в онбординге. Никакого `user_id` «на всякий
случай»: это удорожает каждую фичу.
Триггер для пересмотра — реальное решение о публичном запуске. Тогда план:
таблица `users`, `user_id` во всех таблицах (миграция понятная — все FK уже есть),
PIN → аккаунты, rate-limit и push по пользователю. Объём: ~1–2 недели.

## [11.1] + [17.1] Монолит app.js и полный ререндер → поэтапная распилка

Частично сделано: ререндер сохраняет скролл и фокус; canvas-графики и так
перерисовываются только через rAF после рендера. Полная распилка — самый
рискованный пункт аудита, делать её одним коммитом нельзя.

**Предложение — три безопасных этапа (по этапу за итерацию):**
1. **Вынос библиотек без поведения** (`lib/api.js`, `lib/format.js`, `lib/charts.js`)
   — чистые функции, браузер понимает ES-импорты без сборки. Риск ~нулевой.
2. **Вынос view-функций** (`views/queue.js`, `views/dashboard.js`, …) — каждая
   уже почти чистая функция `state → html`. После этого app.js < 1000 строк.
3. **Точечный ререндер** — каждый view получает контейнер и `update(patch)`;
   фильтры очереди обновляют только `<tbody>`. Делать только для очереди
   (самый длинный список) — остальным экранам полный ререндер не вредит.

## [Раздел 18] Конкурентный анализ → что брать в роадмап

Вердикт аудита разделяю: не догонять YNAB по леджеру, а углублять уникальное ядро
(очередь, trade-off, скоринг, прайс-трекинг). Приоритет заимствований:

1. **«Свободно» как главная метрика** (PocketGuard) — число уже едино по приложению;
   добавить его в push-уведомления и на иконку PWA (Badging API). ~0.5 дня.
2. **Rollover остатка** (YNAB) — при закрытии месяца непотраченный излишек
   предлагается перенести стартовым капиталом в новый план (сейчас он исчезает
   из фокуса). ~1 день, хорошо сочетается с 1.3.
3. **Автоподстановка слоя/категории по названию желания** — словарная эвристика
   («курс», «книга» → рост; «подарок» → забота), не ML. ~0.5 дня.
4. **Отчёты как 3 вопроса** («куда ушло», «что с целями», «тренд излишка») —
   вместо sankey. Делать после 1.3, иначе нечего показывать в «куда ушло».
5. **Импорт CSV-выписки** — только если минимальная версия 1.3 приживётся.
