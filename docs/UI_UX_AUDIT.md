# UI/UX-аудит Capital Queue

Дата аудита: 2026-06-05  
Объект: production `https://salary-planner-475c5d.fly.dev` и текущий код `public/index.html`, `public/styles.css`, `public/app.js`.

## Краткий вывод

Интерфейс уже заметно ушёл от «сырого MVP»: есть цельная визуальная метафора personal finance cockpit, узнаваемая цветовая палитра, сильный первый экран, понятная боковая навигация и хорошая мобильная основа. Главный риск сейчас не в визуале, а в UX-сложности: приложение содержит много сущностей — зарплата, желания, кошельки, инвестиции, ручной план, история, AI — и новичку нужно быстрее объяснять, что делать первым и как связаны вкладки.

Общая оценка: **7.6/10**

| Блок | Оценка | Комментарий |
|---|---:|---|
| Визуальный дизайн | 8/10 | Стиль цельный, современный, с хорошей типографикой и контрастным hero. Есть риск перегруза декоративными слоями и разнородными иконками. |
| Информационная архитектура | 7/10 | Разделы логичные, но их много; «Кабинет», «Кошельки» и «План распределения» могут пересекаться в голове пользователя. |
| Удобство сценариев | 7/10 | Основные сценарии закрыты, но нужен более явный onboarding и пустые состояния с обучением. |
| Мобильный UX | 7.5/10 | Есть bottom-nav, FAB и отдельные mobile rows; нужно сократить число пунктов навигации и проверить длинные таблицы/формы. |
| Доступность | 7/10 | Есть focus-visible, нормальная семантика кнопок и русский язык; нужно улучшить labels, состояния ошибок, контраст вторичного текста и aria для иконок. |
| Доверие/продуктовая ясность | 7.5/10 | Видно, что это личный финансовый cockpit. Не хватает короткого объяснения модели: «сначала стабильные пункты → потом излишки → потом желания». |

## Что работает хорошо

### 1. Первый экран уже продаёт идею

Экран входа выглядит заметно лучше стандартной формы: крупный заголовок, карточка входа, бейдж обновления, спокойная финансовая палитра. Это создаёт ощущение законченного продукта ещё до авторизации.

**Плюсы:**
- сильная визуальная иерархия;
- понятный бренд Capital Queue;
- форма входа не теряется;
- светлый фон и тёплая палитра снижают ощущение «банковской тяжести».

**Что ослабляет:**
- текст в hero сейчас говорит про факт редизайна, а не про пользовательскую ценность;
- блоки «свежие ассеты / новый экран / контрастный кабинет» полезны для проверки деплоя, но не для реального пользователя;
- если это личное приложение, можно говорить проще и конкретнее: «зарплата → обязательное → резерв → инвестиции → желания».

### 2. Дашборд имеет правильную роль

`Кабинет` задуман как главный decision cockpit: пользователь должен быстро понять, сколько денег можно распределить, что безопасно купить и что лучше отложить. Это верная центральная метафора для продукта.

**Плюсы:**
- карточки метрик хорошо сканируются;
- decision cockpit отделён от обычных карточек и выглядит главным блоком;
- есть статусы safe/tight/overallocated;
- распределение по слоям помогает понять структуру зарплаты.

**Риск:** если данных много, экран может стать «панелью всего сразу». Нужно удерживать фокус на одном главном вопросе: *что можно сделать с ближайшей зарплатой?*

### 3. Архитектура функционально полная

Разделы покрывают ключевые сценарии:
- настройка зарплаты;
- добавление желаний;
- приоритизация и дедлайны;
- ручное распределение излишков;
- кошельки месяца;
- инвестиции;
- история решений;
- AI-пояснения.

Это хороший фундамент для личного финансового приложения, особенно под сценарий студента с ограниченной, но регулярной зарплатой.

## Основные UX-проблемы

### P0 — Нет явного onboarding после первого входа

Сейчас пустой кабинет говорит «Сначала настройте будущую зарплату», но не объясняет всю модель продукта. Пользователь может не понять, почему нужно отдельно заполнять зарплату, желания, кошельки и ручной план.

**Рекомендация:** добавить короткий onboarding из 3 шагов на пустом `Кабинете`:

1. **Настрой зарплату** — доход, дата, обязательные расходы, резерв, инвестиции.
2. **Добавь желания** — всё, что хочешь купить, с ценой и приоритетом.
3. **Собери план месяца** — реши, что покупаешь сейчас, а что переносишь.

Хороший формат: горизонтальная checklist-карточка с прогрессом `1/3`, `2/3`, `3/3` и CTA на следующий шаг.

### P0 — Навигация слишком широкая для первого пользователя

Семь разделов + выход в мобильной навигации — много для маленького экрана. При первом использовании пользователь ещё не различает «Кошельки», «План распределения» и «Очередь желаний».

**Рекомендация:** сгруппировать сценарии:

- Основные: `Кабинет`, `Желания`, `План`.
- Вторичные: `Кошельки`, `Инвестиции`, `История`, `AI`, `Данные` — в меню `Ещё` или внутри соответствующих экранов.

Для desktop текущая боковая навигация допустима. Для mobile лучше оставить 4–5 пунктов максимум:
`Кабинет`, `Желания`, `План`, `Инвест`, `Ещё`.

### P1 — Тексты местами описывают реализацию вместо ценности

Пример на первом экране: «Переработал первый экран, навигацию и карточки…» — это полезно для разработчика, но не для пользователя.

**Рекомендация:** заменить продуктовые тексты на пользовательские:

- Было: «План зарплаты, который видно сразу»
- Лучше: «Реши заранее, куда уйдёт следующая зарплата»

- Было: «свежие ассеты без старого кеша»
- Лучше: «сколько останется после обязательного»

- Было: «контрастный кабинет»
- Лучше: «что купить сейчас, что отложить»

### P1 — `Кошельки` и `План распределения` требуют ясного различия

В текущей модели:
- `Кошельки` — карманы под месяц;
- `План распределения` — ручное распределение по желаниям;
- `Очередь желаний` — мастер-список.

Это мощно, но когнитивно тяжело.

**Рекомендация:** добавить на каждый из этих экранов одну строку «Когда использовать»:

- `Очередь желаний`: «Сюда складывай всё, что хочешь купить когда-нибудь».
- `План`: «Здесь решай, что финансировать из ближайшей зарплаты».
- `Кошельки`: «Здесь держи месячные лимиты/карманы, не привязанные к конкретному желанию».

### P1 — Таблица очереди желаний перегружена действиями

В desktop-таблице в строке есть `Trade-off`, `AI почему`, `Копить`, `Редактировать`, `Куплено`, `Удалить`. Это функционально, но визуально шумно.

**Рекомендация:**
- оставить видимыми 1–2 основных действия: `Trade-off` и `Копить`/`В план`;
- остальные действия убрать в меню `⋯`;
- опасное действие `Удалить` показывать только в меню и подтверждать.

### P1 — Нет единого «следующего лучшего действия»

В decision cockpit есть рекомендации, но на уровне приложения не всегда ясно, что делать дальше.

**Рекомендация:** в верхней части `Кабинета` показывать один главный CTA в зависимости от состояния:

- нет плана → `Настроить зарплату`;
- есть план, нет желаний → `Добавить первое желание`;
- есть желания, нет ручного плана → `Собрать план месяца`;
- есть перерасход → `Найти, что перенести`;
- всё ок → `Закрыть месяц` или `Добавить покупку`.

### P2 — Визуальный стиль местами конфликтует между «тёплый editorial» и «технический cockpit»

Палитра сейчас интересная: тёплый бежевый фон + тёмный teal + синие элементы. Это хорошо, но нужно закрепить роли цветов:

- teal/navy — структура и primary action;
- green — положительный финансовый статус;
- amber — предупреждение/проверка;
- red — риск/перерасход;
- beige — фон/спокойствие.

**Рекомендация:** не добавлять новые акцентные цвета без необходимости. Иконки лучше унифицировать: либо emoji-стиль, либо line icons, но не смешивать слишком активно.

### P2 — Пустые состояния должны учить

Сейчас пустые состояния короткие: «Очередь пуста», «Добавьте первый актив». Они работают, но можно сделать их сильнее.

**Рекомендация:** каждое пустое состояние должно отвечать на 3 вопроса:

1. Что это за раздел?
2. Зачем он мне?
3. Что сделать первым?

Пример для `Инвестиции`:

> Здесь ты видишь, сколько уже вложено и как меняется портфель. Начни с первого актива — например, ETF, крипта или депозит.

CTA: `+ Добавить актив`.

### P2 — Мобильный bottom-nav перегружен

На мобильном сейчас видно много вкладок, включая `История`, `AI`, `Выйти`. Это может создать горизонтальный скролл и случайные промахи.

**Рекомендация:**
- убрать `Выйти` из bottom-nav в настройки/меню;
- `Историю` и `AI` переместить в `Ещё`;
- FAB оставить только там, где действие однозначно: на `Желания` — добавить желание, на `Кошельки` — добавить кошелёк; на остальных экранах скрывать или менять подпись.

### P2 — Нужны более явные состояния сохранения и ошибок

Формы и API уже имеют toast/error-логику, но в UX стоит явно показывать:

- успешное сохранение плана;
- ошибку валидации рядом с конкретным полем;
- предупреждение перед закрытием месяца;
- подтверждение перед удалением желания/кошелька/актива.

## Сценарии пользователя

### Сценарий 1: первый запуск

**Цель:** понять, как пользоваться приложением.  
**Текущий путь:** вход → пустой кабинет → настройка зарплаты.  
**Проблема:** не объяснена вся модель.  
**Улучшение:** onboarding-checklist + sample placeholders.

### Сценарий 2: зарплата скоро придёт

**Цель:** решить, что купить и сколько оставить.  
**Текущий путь:** Кабинет → Очередь → План распределения.  
**Проблема:** нужно самостоятельно понять связь между экранами.  
**Улучшение:** CTA из `Кабинета` прямо в следующий шаг + подсказка «сейчас распределяем только излишки».

### Сценарий 3: появилось новое желание

**Цель:** быстро добавить покупку и понять, можно ли её позволить.  
**Текущий путь:** Очередь → quick add → Trade-off.  
**Что хорошо:** quick add расположен правильно.  
**Улучшение:** после добавления показывать мини-результат: «помещается / не помещается / вытеснит X».

### Сценарий 4: пользователь на телефоне в магазине

**Цель:** быстро проверить покупку.  
**Текущий путь:** открыть приложение → найти желание → Trade-off.  
**Проблема:** много вкладок и действий.  
**Улучшение:** быстрый режим `Проверить покупку` на главном экране или FAB с короткой формой `Название + цена`.

### Сценарий 5: конец месяца

**Цель:** закрыть месяц и сохранить историю решений.  
**Текущий путь:** План распределения → Закрыть месяц.  
**Проблема:** действие важное, но может быть неочевидно.  
**Улучшение:** в `Кабинете` показывать CTA `Закрыть месяц`, когда дата зарплаты прошла или план заполнен.

## Рекомендованный roadmap улучшений

### Быстрые улучшения на 1–2 часа

1. Переписать hero copy на пользовательскую ценность, убрать технические формулировки про редизайн/кеш.
2. Добавить `Когда использовать` под заголовки `Очередь`, `Кошельки`, `План`.
3. Скрыть `Выйти` из мобильной bottom-nav и перенести в настройки.
4. Добавить подтверждение удаления желания/кошелька/актива.
5. После quick add показывать toast с понятным next step: `Добавлено. Проверить trade-off?`.

### Средние улучшения на 0.5–1 день

1. Сделать onboarding-checklist для пустого состояния.
2. Добавить главный smart CTA на `Кабинете`.
3. Упростить действия в таблице желаний через меню `⋯`.
4. Улучшить пустые состояния всех разделов.
5. Добавить summary-блок после сохранения плана: зарплата, обязательное, излишки, риск.

### Более крупные улучшения

1. Переработать мобильную информационную архитектуру: `Кабинет / Желания / План / Инвест / Ещё`.
2. Добавить guided flow «Собрать план месяца» в 3 шага.
3. Сделать отдельный быстрый сценарий `Проверить покупку`.
4. Добавить визуальную timeline-логику до зарплаты: сколько дней осталось, что уже профинансировано, что под риском.
5. Добавить lightweight demo/sample mode для первого знакомства без ручного заполнения.

## Приоритетный список конкретных правок

1. **Hero copy**: заменить технические proof-points на value-points.
2. **Empty onboarding**: заменить `noPlanBlock()` на checklist из 3 шагов.
3. **Smart CTA**: вычислять следующий шаг по состоянию данных.
4. **Mobile nav**: сократить пункты и убрать logout.
5. **Queue actions**: свернуть второстепенные действия в `⋯`.
6. **Screen explanations**: добавить короткие help-подзаголовки в `viewQueue`, `viewWallets`, `viewPlan`.
7. **Destructive confirmation**: подтверждать удаление.
8. **Post-add feedback**: после добавления желания предлагать trade-off.

## Итог

Самый высокий ROI сейчас — не в очередном декоративном редизайне, а в снижении когнитивной нагрузки. Визуально приложение уже выглядит как продукт; следующий шаг — сделать его самонаправляющимся: чтобы пользователь всегда видел, где он находится, зачем нужен раздел и какое действие сделать дальше.
