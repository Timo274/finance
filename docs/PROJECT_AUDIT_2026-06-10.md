# Полный аудит проекта — 10.06.2026

Свежий разбор всего репозитория: `server.js` (2138 строк), `src/*` (auth, db, allocation, insights, ai, categories), `public/*` (SPA ~2760 строк, SW, PWA), тесты, CI, деплой на Fly.io.

Общее впечатление: проект в хорошей форме. Архитектура (Express + SQLite + vanilla SPA) адекватна личному инструменту, есть CSP и security-заголовки, rate-limit на вход, JSON-бэкапы, экранирование в UI, unit + e2e тесты, CI с автодеплоем. Ниже — конкретные недоработки (от критичных к мелким) и идеи развития.

---

## 1. Недоработки и баги

### 1.1. `goal_contributions` теряются при backup/restore (данные) — критично
- `exportPayload()` **не включает** таблицу `goal_contributions`, а `restoreFullBackup()` начинает с `DELETE FROM goal_contributions` и ничего туда не восстанавливает.
- Итог: история взносов в цели безвозвратно теряется при любом restore из JSON, хотя сервер её старательно пишет (`/api/items/:id/savings`, закрытие месяца по ручному плану).
- Бонус-наблюдение: таблица **write-only** — её никто не читает и не показывает. Либо добавить в export/restore + вывести историю взносов в карточке цели, либо честно удалить.

### 1.2. `rowToPlan`: `JSON.parse(snapshot)` без try/catch
- В `src/db.js` `rowToItem` парсит `scores` через try/catch, а `rowToPlan` парсит `snapshot` голым `JSON.parse`.
- Один повреждённый snapshot (например, после ручной правки БД или неудачного импорта старого формата) → 500 сразу на `/api/state`, `/api/history` и `/api/allocation` — приложение целиком ложится.

### 1.3. Logout и смена PIN не отзывают сессии
- JWT живёт 30 дней; `logout` лишь чистит cookie у конкретного браузера. Украденный/оставшийся на чужом устройстве токен остаётся валидным.
- Смена PIN (сейчас её и нет как фичи — см. 1.9) тоже не инвалидировала бы старые токены.
- Дёшево чинится: `token_version` в `settings`, класть в payload токена и сверять в `isAuthed()`; logout/смена PIN инкрементит версию.

### 1.4. Рассинхрон версий статики в Service Worker
- `sw.js` precache: `/styles.css?v=20260605-layer1`, а `index.html` подключает `?v=20260606-mobile`. Precache греет мёртвый URL, реальный файл кэшируется только runtime-фоллбеком.
- Версия зашита в 4 местах руками (`CACHE`, `STATIC`, `index.html`, sessionStorage-ключ в `app.js`) — гарантированный источник будущих «у меня старая версия». Стоит генерить одну версию (git short hash) и подставлять её везде при деплое.

### 1.5. Валидация непоследовательна в `/api/items/:id/savings` и `/api/goals`
- `deadline` берётся из body без `isoDateValue()`, `note`/`date` — без `textValue()`/лимита длины, `status` в `/api/goals` принимает любую строку. Везде в остальном API вход аккуратно нормализуется — эти ручки выбиваются.

### 1.6. Закрытие месяца в авто-режиме слишком оптимистично
- Без ручного плана `/api/plan/close` помечает **все** approved-позиции как `bought`, хотя авто-распределение — это рекомендация, а не факт покупки. Плюс в авто-ветке не пишутся `goal_contributions` (в manual-ветке пишутся).
- См. идею 2.4 (чек-лист «что реально купил» при закрытии).

### 1.7. Валюта и цены: упрощения, которые дадут неверные цифры
- `valuationForAsset`: цена из Yahoo приходит в валюте тикера (не обязательно USD — EUR/GBP-бумаги), но всегда умножается на USD/UAH-курс. Для актива с `currency: "UAH"` rate=1, т.е. цена USD-тикера записывается как гривны.
- Курс 43.5 захардкожен как дефолт и обновляется только руками. Идея: автокурс с NBU API (`bank.gov.ua/NBUStatService/...`, бесплатно, без ключа) раз в сутки.
- `CG_MAP` — 24 захардкоженных коина; для неизвестного тикера цена молча не обновится (`_meta.errors` — только счётчик, UI не говорит, какие именно тикеры не нашлись).

### 1.8. Рост таблицы `asset_valuations` без дедупликации
- Каждый «обновить цены» вставляет новые строки valuations; `getPortfolio()` каждый раз тащит **все** транзакции и **все** оценки в память и фильтрует JS-ом. При ежедневном использовании за пару лет это тысячи строк на каждый `/api/state` (а он дёргается при каждом poll-изменении).
- Дёшево: upsert по `(asset_id, date)` вместо insert + индексы `asset_transactions(asset_id)`, `asset_valuations(asset_id, date)`, `items(status)`.

### 1.9. Нет смены PIN
- Единственный способ сменить PIN — руками удалить `pin_hash` из БД. Стоит добавить `/api/auth/change-pin` (старый PIN → новый) + пункт в настройках.

### 1.10. Лимит body 1 МБ против растущего бэкапа
- `express.json({ limit: "1mb" })` глобальный. Полный JSON-export с историей valuations со временем превысит 1 МБ → restore собственного бэкапа станет невозможен. Поднять лимит точечно для `/api/import` (5–10 МБ).

### 1.11. Бэкапы лежат на том же volume, что и БД
- Смерть volume на Fly = потеря и базы, и всех бэкапов одновременно. Минимум — проверить, что включены Fly volume snapshots; лучше — offsite-копия (см. идею 2.2).

### 1.12. Гигиена кода и мелочи
- `server.js` (2138) и `public/app.js` (2761) — монолиты; назрело разбиение: `routes/{auth,items,plan,investments,backup,ai}.js`, на фронте — модули по вкладкам.
- Legacy-слой: `investment_accounts`/`investment_updates`, fallback-чтения из `settings` JSON (`getInvestments`, `getWallets`, `getManualPlan`), legacy `POST /api/investments` с полной перезаписью. После миграции это мёртвые ветки — наметить версию, в которой их выпилить (export уже `version: 4`).
- Нет `Strict-Transport-Security` заголовка (force_https на Fly есть, HSTS — нет).
- `UNIQUE(plan_id, item_id, source)` при `plan_id IS NULL` в SQLite не работает как upsert (NULL ≠ NULL) — сейчас спасает delete-then-insert, но импорт может насоздавать дублей решений с NULL plan_id.
- Нет ESLint/Prettier ни локально, ни в CI — для двух файлов на ~5к строк линтер уже окупится.
- `npm audit --audit-level=moderate` в CI может внезапно красить билд из-за чужой уязвимости без вашего изменения — лучше вынести в отдельный allow-fail job или еженедельный cron.
- Поллинг `/api/version` каждые 5 с держит Fly-машину постоянно живой при открытой вкладке — `auto_stop_machines` фактически не срабатывает. Вариант: увеличивать интервал при неактивности (30–60 с после 5 минут без действий) или перейти на SSE.

## 2. Идеи развития (по убыванию ценность/усилия)

### Быстрые победы
1. **Автокурс UAH/USD** с NBU API раз в сутки + бейдж «курс от {даты}». Убирает ручную рутину и хардкод 43.5.
2. **Offsite-бэкап**: после ночного JSON-снапшота заливать его в приватный GitHub-репо/gist или S3/R2 (токен в env). 20 строк кода — и данные переживают смерть volume.
3. **История взносов по цели** — UI поверх уже пишущейся `goal_contributions` (после фикса 1.1): мини-таймлайн «когда и сколько докинул».
4. **Чек-лист при закрытии месяца**: показать approved-список с чекбоксами «реально купил» → купленное в `bought`, остальное остаётся в очереди с накоплением. Закрывает и недоработку 1.6.
5. **Смена PIN** (1.9) + кнопка «выйти на всех устройствах» (1.3).

### Среднее
6. **График динамики**: net worth (portfolio valuations по датам) и «свободные излишки по месяцам» (данные уже есть в snapshot закрытых планов). Сейчас история — текст, а тренд — главная мотивация копить.
7. **Повторяющиеся желания/подписки**: флаг `recurring` → при закрытии месяца позиция автоматически возвращается в очередь на следующий месяц.
8. **Напоминания**: PWA push или Telegram-бот — «завтра payday, закрой месяц и разложи зарплату», «дедлайн „X“ через 5 дней». Один cron на сервере + sendMessage.
9. **Monobank personal API**: подтянуть фактические траты месяца и показать план/факт (где «утечки» съели больше, чем планировал). Ключ — личный токен в env, без OAuth.
10. **Цена-линк у желания**: хранить URL, периодически проверять цену и алертить «подешевело на 15%» / обновлять стоимость в очереди.

### Дальше
11. **Сценарий «что если»**: слайдеры зарплата/обязательные/резерв с живым пересчётом очереди — наглядно «что будет, если зарплата вырастет до 30к».
12. **AI как ритуал, а не вкладка**: ежемесячный авто-разбор при закрытии месяца («что пожалеешь», «что мусор в очереди») + кнопка «отговори меня» в карточке. Реальный стриминг вместо имитации в `/api/ai/chat/stream` (сейчас отдаёт весь ответ одним куском).
13. **Годовой отчёт**: Excel/CSV по закрытым месяцам — сколько ушло по слоям, сколько желаний куплено/брошено, динамика накоплений.
14. **Мультивалютные желания**: цена в USD/EUR с автоконвертацией по текущему курсу (актуально для техники).

## 3. Что осознанно НЕ предлагаю
- Multi-user, регистрация, PostgreSQL, микросервисы — для личного инструмента это шаг назад.
- Переписывание фронта на фреймворк — vanilla SPA работает; модульность важнее смены стека.
- Реалтайм-вебсокеты — поллинг + BroadcastChannel для одного пользователя достаточно (макс. — SSE ради экономии Fly-часов).

## 4. Рекомендуемый порядок
1. Фиксы данных: 1.1 (contributions в export/restore), 1.2 (try/catch snapshot), 1.10 (лимит import).
2. Безопасность: 1.3 (token_version), 1.9 (смена PIN), HSTS.
3. Гигиена: 1.4 (единая версия статики), индексы/дедуп valuations (1.8), валидация (1.5).
4. Быстрые победы из идей: автокурс, offsite-бэкап, чек-лист закрытия месяца.
5. Потом — графики, recurring, напоминания.
